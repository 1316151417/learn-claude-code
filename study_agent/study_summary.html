技能地图（面向有经验后端开发者）

1. LLM 与 Agent 基础

  • 理解 LLM Agent 核心范式：思考 → 调用工具 → 观察结果 的 ReAct 循环
  • 掌握四种典型架构：
    • ReAct：Thought / Action / Observation 显式循环
    • Plan-then-Execute：Planner 制定计划 + Executor 执行
    • 多 Agent：Orchestrator–Worker 分工协作
    • 图/状态机编排：有向图 + 条件分支，高可控性
  • 核心理念：将 Agent 系统视为「带状态机和消息流的微服务系统」

2. 主流框架与生态

  • LangChain：老牌 LLM 应用框架，生态最大
  • LangGraph：图式编排，适合复杂有状态工作流
  • AutoGen（微软）：多 Agent 会话 + 异步消息传递
  • CrewAI：角色化团队协作，易上手
  • Google ADK：云原生，2025年新推出

建议选 Python + LangChain/LangGraph 为主线

3. 工具调用、RAG 与记忆

  • 工具调用：HTTP API、数据库查询、代码执行作为 Tools
  • RAG：文档分块 → 向量嵌入 → 向量库检索 → 喂给 LLM
  • 记忆：短期（对话历史）+ 长期（知识库、用户偏好）

4. Agent 工程 vs 传统后端

  • 可靠性：失败重试、幂等、状态可追踪
  • 成本控制：Token 用量、大小模型搭配
  • 监控评估：Thought/Action 轨迹、链路分析
  • 将微服务的认证/限流/Tracing 直接搬到 Agent 服务

🧠 二、理论基础

2.1 ReAct (Reason + Act) 范式

ReAct 强制模型交替进行逻辑推理与工具调用来与外部世界交互，逐步向解决问题迈进。

Agent 组件

组件说明
LLM（大脑）生成决策步骤，释放思想和行动指令
Tools（工具）外部函数或 API（搜索、计算、数据库操作等），以名称+自然语言描述注册
Agent Executor执行模型的行动决定，调用工具并返回结果
State（状态）累积的对话和行动历史，随步骤不断更新

工作循环

1. 接收请求 (Query)
↓
2. 生成推理和行动 (Think)
Thought: 内部推理 → Action: 调用工具
↓
3. 执行操作 → 调用工具获取结果
↓
4. 状态更新 (Observation) → 添加到历史
↓ 循环
5. 完成END → 返回 Final Answer

内部反思（Thoughts）

  • 分析和规划：「先收集数据，再识别模式，最后准备报告」
  • 自我反思：「之前的方法无效，我将尝试不同策略」
  • 记忆整合：「用户之前说偏好 Python，用这种语言提供示例」
  • 优先级排序：「添加新功能前，先消除安全漏洞」

LangGraph 中的实现细节

  • 使用 tool-calling (function-calling) 替代原始文本 prompt 解析
  • 使用参数数组调用工具（而非单个大字符串）
  • 允许一次调用多个工具
  • 默认不显式生成 Thought 阶段（LLM 能力已提升，不再必要）

优缺点

✅ 优点

  • 灵活的逐步解决方案，可动态解决复杂问题
  • 易于开发，许多框架提供现成实现
  • 通用性强，适用于各种领域

❌ 缺点

  • 每步都需要 LLM 调用，长链缓慢且昂贵
  • 没有全局计划，可能进入死胡同
  • 工具/历史过多时，可扩展性有限

其他 Agent 构建方法

Chain-Of-Thought (CoT) Tree-Of-Thought (ToT) Plan-and-Execute ReWOO LLMCompiler Reflexion Self-Discover MRKL Systems PAL Toolformer

多 Agent 系统：Network / Supervisor / Hierarchical

2.2 AI 智能体从第一性原理理解

定义

AI 智能体不是普通 LLM，而是能独立感知、推理、决策、调用工具、执行行动的闭环系统。

特性标准 LLMAgent
功能文本到文本映射拥有行为循环：感知→推理→决策→行动→反馈→再推理
工具能力无/有限可主动调用
自主行动否是
循环反馈否是

能力分级

Level 0：标准 LLM
输入文本 → 输出文本，只能依赖内部知识
↓ + 工具调用
Level 1：LLM + Tool Use
生成结构化 API 调用，结果反馈给 LLM
↓ + 问题分解
Level 2：问题分解
任务分解 + 迭代执行 + 反馈循环
↓ + 自主决策
Level 3：高自主性 Agent
长期运行、自主决策 next action、后台持续运行

两大基本支柱

工具使用

LLM 知识有时效性限制和幻觉风险，需要调用更可靠的外部工具（搜索 API、数据库、计算器等）来克服限制。

流程：停止生成 → 解析调用 → 执行工具 → 结果放入上下文 → 继续生成

推理能力

Agent 需要拆解问题、设计计划、决定调用哪些工具、理解结果并调整计划。

专门的推理模型能生成更长、条件适应性强的思维链（reasoning trajectory）。

ReAct：底层行为循环

Thought → ...

3.1 四种核心 Agent 架构

基于视频《Designing AI Agents》的核心知识总结。六项评估指标：推理能力、成本、可靠性、延迟、可扩展性、可解释性。

🎭 策略一：即兴者 (ReAct)

原理：Thought → Action → Observation 持续循环

成本：线性增长（每步都调用 LLM），100万任务/天 ≈ $40万/天

最适合：客户支持、实时交互、探索性研究

灵活性最高成本高

📐 策略二：蓝图 (Plan-Execute)

原理：规划者（昂贵模型 GPT-4）→ 执行者（便宜模型 Haiku）

优势：昂贵模型只调用一次，成本效率高

最适合：结构良好的问题、成本敏感应用

成本最低系统脆弱

👥 策略三：专家团队 (Hierarchical)

原理：Manager Agent + 多个 Specialist Agent

优势：专业化 + 并行处理 + 可扩展

最适合：企业级复杂系统、并行化场景

可扩展性最强协调开销大

🔄 策略四：迷宫 (Graph)

原理：有限状态机 — State + Node + Edge

优势：可靠性最高、可恢复、支持人工介入

最适合：长期运行流程、金融交易、需审批的工作流

可靠性最高灵活性差

3.2 2025年 Agentic AI 框架对比（五大框架）

LangChain Agents

最大社区（11.6万⭐），ReAct 风格 agents，与 LangChain 工具无缝集成。高级别抽象但精细控制能力较弱。

LangGraph

基于图的 API（1.92万⭐），每个计算是节点，边控制流程。集成了 LangSmith 用于调试、持久执行、人在回路。适合复杂有状态应用。

AutoGen（微软）

多智能体对话框架（5万+⭐），以代码为中心的 API，异步事件驱动系统。强大的「智能体委托」模型。v0.7 增加了流式支持。

CrewAI

强调简洁性（3.8万+⭐），引入 Crews（智能体团队）和 Flows（逐步工作流）。清晰的对象模型。10万认证用户，每周更新。

Google ADK

2025年4月推出（1.3万⭐），代码优先、模型无关。原生支持 A2A 通信协议。与 Vertex AI 深度集成。云原生设计。

4.1 Quick Start — 天气预报 Agent

01_quick_start.py完整版

演示 LangChain create_agent 构建完整 Agent，包含工具调用、结构化输出、上下文传递。

核心特性

  • 自定义 Context：通过 context_schema=Context 传递用户上下文（如用户名）
  • 结构化输出：response_format=ToolStrategy(ResponseFormat) 定义双关语+天气格式
  • 工具链：get_user_city → get_weather_for_city → ResponseFormat
  • System Prompt：定义角色（天气预报专家）、工具调用流程、示例对话

工具定义

@tool
def get_user_city(runtime: ToolRuntime[Context]):
    """获取用户所在城市"""
    user_name = runtime.context.user_name
    city = "北京市" if user_name == "周杰" else "四川市"
    return {"city": city}

@tool
def get_weather_for_city(city: str) -> str:
    """获取城市的天气"""
    return f"{city} 总是阳光明媚!"

Agent 构建

agent = create_agent(
    llm,                          # 智谱 GLM 模型
    system_prompt=SYSTEM_PROMPT,  # 角色 + 工具流程说明
    tools=[get_user_city, get_weather_for_city],
    context_schema=Context,       # 上下文结构
    response_format=ToolStrategy(ResponseFormat),  # 结构化输出
    checkpointer=checkpointer    # InMemorySaver 持久化
)

对比版本：无结构化输出

01_quick_start_error.py精简版

与完整版相比，移除了 response_format 参数，展示不带结构化输出的 Agent 行为差异。

4.3 Middleware — 人工介入审批

03_middleware.pyHuman-in-the-Loop

使用 HumanInTheLoopMiddleware 在工具执行前暂停，等待人工审批。

中间件配置

middleware=[
    HumanInTheLoopMiddleware(
        interrupt_on={
            "get_weather_for_city": {
                "allowed_decisions": ["approve", "edit", "reject"],
            },
            "get_user_city": False  # 不中断
        }
    )
]

三种审批操作

approve（批准）

Command(resume={"decisions": [{"type": "approve"}]})

edit（编辑）

Command(resume={"decisions": [{
    "type": "edit",
    "edited_action": {"name": tool_name, "args": new_args}
}]})

reject（拒绝）

Command(resume={"decisions": [{"type": "reject"}]})

5.9 Functional API — 函数式工作流

09_functional.py新范式

与 Graph API 不同，用标准 Python 控制流（if/for）定义工作流，无需构建 State/Node/Edge。

核心概念

@task

封装离散工作单元，结果可被 checkpoint 持久化。恢复时不重算。

@task
def generate_topic_essay(topic: str) -> str:
    time.sleep(0.5)
    return f"关于「{topic}」的短文"

@entrypoint

工作流入口，管理执行流、中断恢复、短期记忆。

@entrypoint(checkpointer=InMemorySaver())
def essay_workflow(topic: str) -> dict:
    essay = generate_topic_essay(topic).result()
    is_approved = interrupt({"essay": essay})
    return {"essay": essay, "is_approved": is_approved}

短期记忆演示（previous 参数）

@entrypoint(checkpointer=InMemorySaver())
def accumulator(number: int, *, previous: int = None) -> int:
    previous = previous or 0
    return number + previous

# 累加链: invoke(1)→1, invoke(2)→3, invoke(10)→13
