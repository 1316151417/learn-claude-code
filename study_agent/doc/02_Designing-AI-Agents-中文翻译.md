## 为什么架构很重要

当我们谈论 agents 时，我们不再只是谈论发送到大语言模型的简单提示。我们谈论的是为实现长期目标而设计的复杂、自主的系统。

要成功构建这些系统，您需要的不仅仅是一个强大的 LLM。您需要一个坚实的架构蓝图。

这就好比城市规划或设计一座巨大的摩天大楼（背景图中所反映的）。您需要理解决定 agent 如何推理、如何行动以及最终如何在现实世界中表现的基本结构。

---

## 架构是战略必要性

使命很明确：**我们必须理解为什么架构在现代 AI agent 开发中不是可选项。**

许多开发者将 agents 视为 glorified chatbots（升级版聊天机器人），纯粹专注于 prompt engineering。但是，没有计划构建的系统与具有坚实架构基础构建的系统之间的差异是巨大的。

这不仅仅是 10% 的改进。这是大规模成功与失败之间的区别。它区分了可维护的解决方案和完全的调试噩梦。也许最重要的是，它将有利可图的系统与在不必要的 API 调用上花费巨额资金的系统区分开来。

您的架构选择是一个战略决策，决定了 agents 性能的每一个关键方面。

---

## 六个关键评估维度

我们使用六个关键维度来评估任何 agent 架构，本质上是成功的记分卡：

1. **推理能力**：Agent 如何思考？是仔细的逐步过程，还是试图一次性做所有事情？

2. **成本**：这是最重要的。我们主要通过 LLM 调用次数和由此产生的 token 使用量来衡量。

3. **可靠性**：系统能否处理意外路径，还是在面对新事物时频繁崩溃或出错？

4. **延迟**：响应速度有多快？这对于面向用户的应用程序至关重要。

5. **可扩展性**：当从处理一个任务扩展到同时处理数百万个任务时，系统表现如何？

6. **可解释性**：您能否理解为什么 agent 做出了特定决策，这对于审计和合规至关重要？

这六个指标构成了评估我们今天讨论的每个策略的基础。

## 四种核心架构策略

为了理解这些 agent 架构，让我们使用一个类比。想象您正在领导一个复杂的项目，比如组织高风险行动或复杂的银行抢劫。您选择的策略决定了从团队规模到通信协议的所有内容。

同样的原则也适用于 agents。**Agent 架构是系统思考和行动的基本蓝图。**

我们已经确定了四种核心"抢劫"策略，每种都代表一个主要的架构模式：

### 策略一：即兴者 (The Improviser)
通常称为 **ReAct 风格**，这个 agent 逐步适应，不断根据新信息做出反应。它高度灵活，但由于需要不断重新计算，可能缓慢且不可预测。

### 策略二：蓝图 (The Blueprint) 或"计划然后执行"
您使用强大的模型预先创建详细、僵化的计划，然后更便宜的模型精确地遵循它。它高度高效但脆弱。如果情况在执行中途发生变化，整个计划可能变得无效。

### 策略三：专家 (The Specialists)
对应**分层架构**。您组建一个团队，将任务委托给各种专家，并使用 manager agent 协调他们的工作。它对于大规模问题扩展性非常好，但需要大量的协调和通信逻辑开销。

### 策略四：迷宫 (The Labyrinth) 或基于图的方法
您设计一个详细的流程图，具有明确的决策点、条件边和定义的状态。它高度确定且可恢复，提供巨大的可靠性，但它要求架构师预先预测 agent 可能采取的每条可能路径。

理解这四种核心策略是成为有效 agent 架构师的第一步。

---

## 策略一详解：即兴者 (ReAct)

ReAct 代表 **Reasoning plus Acting（推理加行动）**。这是最直观的模式，因为它通过思想、行动和观察的连续反馈循环来镜像人类解决问题。

**ReAct 的力量**在于其自然适应。反馈循环使 agent 能够在工具失败或出现意外信息时立即调整其下一个思想和行动。它真正灵活。

**然而，代价是**即兴创作的巨大成本。每一个循环（涉及思想、行动和由此产生的观察）都需要一次 LLM 调用。这使得成本与任务复杂性成线性关系。需要 10 个循环的任务是单次调用成本的 10 倍。

在大规模上，这很快变得不可持续。例如，每天仅处理 100 万个任务，每个需要 8 次调用，每天可能花费 40 万美元，凸显了优化的需要。

**尽管有成本，ReAct 最适合**：
- 实时适应和动态环境
- 交互式场景，如客户支持
- 探索性任务，如研究

**本质上，当灵活性和响应性比最小化 API 成本更重要时使用 ReAct**，例如当您需要有效地导航未知环境时。

---

## 策略二详解：蓝图 (计划然后执行)

此策略将 agent 分为两个不同的角色，专门针对成本和鲁棒性进行优化。我们在规划者和执行者之间分配劳动。

**规划者**是一个强大的、昂贵的 LLM（也许是 GPT-4 或类似模型），仅在前面使用一次来执行高级战略思考。其输出是详细的逐步计划。

**执行者**相比之下是一个更简单、更便宜的模型（如 Claude 3 Haiku），执行战术执行。它一次一步地遵循计划，而不需要推理整体策略。

**这种方法的力量是三重的**：
1. **巨大的成本效率**：昂贵的模型只调用一次
2. **鲁棒性**：计划提供了清晰的审计跟踪
3. **我们最大化使用模型的能力**：优化强大的模型用于规划，更便宜的模型用于可预测的顺序行动

**然而，代价是系统脆弱**。如果步骤在执行中途失败，整个计划可能无效。这需要一个复杂且非平凡的重新规划循环，这通常会抵消最初的效率收益。

**因此，"计划然后执行"最适合**：
- 结构良好、可分解的问题
- 成本关键应用，其中最小化 token 使用是关键
- 具有清晰依赖关系的多步骤任务
- 任何行动序列的可审计性对合规或调试很重要的情况

---

## 策略三详解：专家 (分层 Agents)

策略三通过引入专家或分层 agents 来解决单个 agent 的固有限制，如认知负荷和性能瓶颈。

此架构创建一个多层系统，包括 manager agents 和 specialist agents。

**Manager agent** 接收主要目标，分解它，将子任务委托给专家，最后将他们的输出合成为最终答案。

**Specialists**（如研究 agent、数据分析 agent 或写作 agent）每个都针对其特定领域进行了优化。

**这里的力量源于**：
1. **专业化**：允许每个 agent 针对其特定工作使用不同的模型、提示或工具集进行微调
2. **并行化**：意味着多个专家可以同时工作，与顺序 ReAct 循环相比显著降低延迟
3. **可扩展性**：系统自然扩展以处理巨大问题，只需添加更多专家或新的管理层

**代价是协调开销**。管理这个团队需要复杂的逻辑来选择专家、委托策略和强大的 agent 间通信。这种增加的复杂性由于通信轮次引入了潜在延迟，并使系统最初更难调试。

**分层 agents 最适合**：
- 企业规模复杂性
- 具有清晰可分离领域的问题
- 绝对需要并行化以获得最大性能的情况
- 需要清晰治理和专门行动审计跟踪的大型系统

---

## 策略四详解：迷宫 (基于图的编排)

我们最后的架构方法是策略四，迷宫或**基于图的编排**。

此方法将整个 agent 系统视为有限状态机，由明确的状态、节点和条件边定义。此设计启用确定性和完全可恢复的工作流。

架构依赖于三个核心组件：

**节点**：执行动作的函数、工具或 agents

**状态**：贯穿整个工作流持续的共享数据结构，保存所有中间结果

**边**：基于当前状态定义流的条件连接

**力量（我们称之为踢）是无与伦比的可靠性**。因为状态是明确且受管理的，工作流是可能的且可恢复的。这对于长期运行的流程和工作流至关重要，这些流程和工作流需要人在回路交互，如批准步骤。

它提供了无与伦比的可靠性，因为您总是确切地知道您在哪里以及如何重新开始。

**然而，代价是架构师的负担**。架构师必须预先预测并定义所有可能的状态和转换。这牺牲了您在 ReAct 中发现的自发适应性，因为 agent 受预定义图流的约束。

**基于图的编排最适合**：
- 复杂分支和重型条件逻辑
- 长期有状态流程
- 固有需要人工参与的工作流
- 高风险生产系统，其中可靠性和可审计性绝对至关重要，如金融交易处理

---

## 第五种技术：Tool Fitter

在比较四种核心架构之前，我们需要解决第五种技术，**伪造者的直觉或工具适应器**。

这不是架构，而是一种不同的工具利用技术，通常与架构选择共存。

在 ReAct 中，agent 基于推理决定使用工具。这是一个明确的指令。

**使用 Tool Fitter**，LLM 被微调以在其文本输出中直接生成工具调用，隐式地学习工具何时会改善其响应。

**这种方法的力量**在于学习的工具使用。模型从数据中隐式地学习如何使用工具，而不是通过明确的提示机制。这可能是高度有效的，因为工具调用无缝且直接地集成到生成流中。LLM 本质上将工具使用伪造到其输出中。

**然而，代价是高进入门槛，需要重新训练**。这种方法从根本上需要微调底层 LLM 模型本身。它不能应用于仅 API 的预训练模型（如 GPT-4），在那里您无法访问权重。

此外，添加新工具或显著更改现有工具通常需要完全重新训练模型。

**Tool Fitter 最适合**：
- 您可以微调自己的模型
- 工具使用是学习行为而不是指示行为
- 您有稳定且不变的工具集
- 专注于特定任务的专业领域模型，如复杂计算或信息检索

## 四种架构对比总结

让我们使用之前定义的六个指标进行五种策略的关键权衡比较：

| 指标 | ReAct | 计划然后执行 | 分层 Agents | 图编排 | Tool Fitter |
|------|-------|-------------|------------|--------|------------|
| **成本效率** | 低（每步都昂贵） | 高 | 中 | 中 | 高 |
| **适应性** | 优秀 | 受限 | 受限 | 受限 | 受限 |
| **推理透明度** | 高 | 高 | 高 | 高 | 低 |
| **可扩展性** | 低 | 中 | 优秀 | 中 | 中 |
| **实现复杂性** | 低 | 中 | 高 | 高 | 高 |

### 关键结论

- **ReAct 在适应性和简单性方面获胜**
- **计划然后执行和 Tool Fitter 在成本效率方面获胜**
- **分层 Agents 在可扩展性方面获胜**
- **基于图的编排在可靠性方面获胜**

这些不是客观排名。它们是基于什么约束对您的使命最重要的战略选择。

## 架构师的选择：如何将策略与使命匹配

我们可以使用这个流程图作为决策指南：

### 决策流程

从左上角开始：

**问题 1**：您的环境动态且不可预测吗？
- **否** → 移至优化结构和成本
- **是** → 移至下一个问题

**问题 2**：您是否首先需要实时适应？
- **是** → 使用 **ReAct**
- **否** → 继续流程

**问题 3**：问题结构良好且可分解吗？
- **是** → 移至成本问题
- **否** → 移至分层路径

**问题 4**：成本最小化是主要驱动力吗？
- **是** → 使用**计划然后执行**
- **否** → 继续

**问题 5**：您需要管理具有许多专门任务的极端复杂性吗？
- **是** → 并行化有益吗？
  - **是** → 使用**分层 Agents**

**问题 6**：工作流是否长期运行，具有复杂分支并需要人工输入？
- **是** → 生产可靠性和可恢复性关键吗？
  - **是** → 使用**基于图的编排**

**对于 Tool Fitter**：
- 您可以微调自己的模型吗？
- 工具使用是您想要嵌入的学习行为吗？
- **两者都是** → 使用 **Tool Fitter**

遵循这个逻辑路径确保您的架构选择与您特定用例的性能约束完美对齐。

## 框架选择

选择架构仅仅是第一个战略步骤。接下来您需要合适的工具，构建它的武器库。

2025 年的框架格局由几个关键参与者主导，每个都有独特的哲学和一套专门的优势。

我们将检查六个最突出的框架：
1. **OpenAI Agents SDK**：提供极简路径
2. **LangChain + LangGraph**：终极灵活性
3. **AutoGen**：专注于对话式多 agent 系统
4. **CrewAI**：直观的基于角色的团队合作
5. **Semantic Kernel**：微软的企业级编排工具
6. **n8n**：视觉自动化工具，非常适合非技术用户和人在回路工作流

### 工程师工具包

这些框架通常受喜欢低级代码和最大定制的开发者青睐。

#### 1. OpenAI Agents SDK
- **哲学**：极简主义，专注于四个核心原语，挑战您自己构建其余部分
- **最适合**：简单的 ReAct 循环和快速原型，特别是如果您专门使用 OpenAI 模型
- **优势**：简单性，最容易学习，与 OpenAI 生态系统紧密集成
- **弱点**：缺乏原生图编排或托管内存，限制复杂工作流

#### 2. LangChain + LangGraph
- **哲学**：成为综合生态系统，旨在解决您遇到的任何 LLM 问题
- **最适合**：需要强大的基于图的状态机的复杂、生产级系统
- **优势**：极端灵活性，出色的图支持，卓越的调试能力
- **弱点**：陡峭的学习曲线，对于简单或轻量级任务通常过度

#### 3. AutoGen
- **哲学**：Agents 是通过消息交互的对话实体，促进它们之间的自然通信
- **最适合**：多 agent 研究、模拟和协作工作流
- **优势**：多 agent 协作的卓越性，高度自然的基于对话的范式
- **弱点**：不太适合严格的确定性工作流，与 LangGraph 相比状态管理更有限

### 团队和企业工具包

这些框架优先考虑结构化流程、治理和可访问性。

#### 4. CrewAI
- **哲学**：构建 AI 团队，每个 agent 都有特定角色并协作实现目标
- **最适合**：基于角色的团队合作、内容创作以及新进入 agentic 开发的团队
- **优势**：直观的基于角色的设计，高度初学者友好，快速实现
- **弱点**：在预定义团队结构之外构建高度复杂或定制的架构时灵活性不如 LangChain

#### 5. Semantic Kernel
- **哲学**：专注于企业级编排，具有高度结构化和可预测的流程
- **最适合**：企业组织，特别是与微软 Azure 生态系统对齐的，以及需要特定治理的结构化工作流
- **优势**：深度 Azure 集成，使用依赖注入模式促进非常干净的架构
- **弱点**：学习曲线比 CrewAI 更陡峭，对复杂的松散耦合多 agent 系统的原生支持较弱

#### 6. n8n
- **哲学**：视觉可访问的自动化，为非技术团队设计
- **最适合**：商业用户、一般工作流自动化以及需要人在回路干预的流程
- **优势**：节点视觉界面，对非开发者容易，超过 500 个连接器的强大集成生态系统
- **弱点**：不太适合复杂的深度代码级 agent 推理或高度抽象的任务

## 真实世界场景应用

让我们通过将理论应用于四个真实世界使命场景来巩固我们的知识。

### 场景一：客户支持聊天机器人

**需求**：实时交互响应性和高适应性

**理想架构**：**ReAct**，专门选择其在处理不可预测的用户请求方面的灵活性

**最佳框架**：**OpenAI Agents SDK** 或 **CrewAI**，两者都为构建强大的、自适应的对话循环提供速度和简单性

### 场景二：数据分析管道

**需求**：多步骤执行、成本敏感、需要高可靠性

**架构**：必须结合效率和鲁棒性。我们选择**计划然后执行**以获得前期效率，但关键是在图上运行以获得可靠性。

**最佳框架**：**LangChain + LangGraph** 或 **Semantic Kernel**，两者都提供必要的工具来管理多个步骤和依赖项之间的状态的强大确定性流程。

### 场景三：研究团队模拟

**需求**：多个专家、协作、深度探索

**理想架构**：**分层**，选择其专业化能力

**最佳框架**：**AutoGen** 或 **CrewAI**，两者都在管理和协调多 agent 协作方面表现出色

### 场景四：企业保险索赔工作流

**需求**：复杂分支、强制性人工批准步骤、清晰的审计跟踪、通常由非技术用户操作

**理想架构**：**基于图**，选择其人在回路流程的有状态管理及其固有的可审计性

**最佳框架**：**n8n**，提供视觉无代码工作流构建器，可容纳复杂的流程控制和广泛的集成需求。非常适合大规模企业自动化。

## 结论：架构是战略

架构是战略。至关重要的是要将 agentic 架构不仅仅是学术概念。它们是本质上产生权衡的战略选择。

作为架构师，您的主要工作是有效地理解和导航这些选择。

我们可以将这些权衡可视化为四个关键象限：

1. **确定性与灵活性**：您优先考虑可预测可靠的系统，还是自适应且令人惊讶的系统？

2. **延迟与推理深度**：您需要快速廉价的响应，还是缓慢、资源密集但高质量的推理输出？

3. **成本与质量**：您目标是仅仅足够的预算友好解决方案，还是真正优秀的昂贵系统？

4. **控制与自主**：您需要人在回路安全和治理，还是旨在完全自主且可能风险更高的操作？

### 实践者的思维模式

成功构建 agents 需要采用实践者的思维模式，一个四步过程：

**步骤 1**：**理解问题** - 预先定义您的主要约束，如成本、质量、速度或可靠性

**步骤 2**：**匹配架构** - 选择蓝图（ReAct、计划然后执行、分层或图），最好与那些定义的约束对齐

**步骤 3**：**选择框架** - 选择工具（LangChain、AutoGen 或 Semantic Kernel），最适合您团队的技能和您选择的架构

**步骤 4**：**原型、验证、迭代** - 构建系统，根据六个关键指标衡量其性能，并完善设计

这种系统化方法是偶然构建 agents 与有目的和精确地架构真正智能之间的区别。
